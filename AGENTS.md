# AGENTS.md

## Scope
- This repo is a standalone World of Warcraft addon (`Bartender4`) for TBC-era API (`## Interface: 20400` in `Bartender4.toc`).
- There is no build/test/lint pipeline in-repo; verification is in-game.

## Load Order and Wiring (Critical)
- `Bartender4.toc` is the source of truth for addon load order; keep it in sync when adding/removing top-level files.
- XML files (`embeds.xml`, `locale/locale.xml`, `actionBar/ActionBars.xml`, etc.) define intra-module script load order; add new Lua files to the correct XML include.
- Core boot order is: embedded libs -> locales -> `Bartender4.lua` -> `Options.lua` -> prototypes -> modules.

## Architecture Map
- Core addon object and profile defaults: `Bartender4.lua`.
- Global options and slash command registration (`/bar`, `/bt`, `/bt4`): `Options.lua`.
- Shared bar behavior: `barPrototype/` and `buttonBarPrototype/`.
- Main action bars module: `actionBar/`.
- Special bars (bag, micro menu, pet, stance): `specialBars/`.
- Button implementations: `buttonPrototypes/`.

## Localization Workflow
- `locale/enUS.lua` is generated (header says "Generated by Babelfish script, do not edit directly").
- Regenerate locale key lists with `locale/Babelfish.lua` instead of hand-maintaining `enUS.lua` keys.
- Locale files must be UTF-8 **without BOM** (explicitly required by `locale/Babelfish.lua`).

## Editing Conventions That Matter Here
- Prefer preserving existing style: tabs for indentation, legacy Lua idioms, Ace3 module patterns.
- Do not modernize APIs for retail WoW; this code targets the older client API surface.
- Avoid editing vendored libraries under `libs/` unless the task is explicitly about dependency/vendor code.

## Verification
- Manual verification path: copy/update addon in AddOns folder, reload UI, open config via `/bt4` (or `/bt`, `/bar`).
- Test lock/unlock, bar movement, state switching, and keybinding reassignment (module handles combat lockdown checks).

## Commit Messages
- Use Conventional Commits format (`type(scope): subject`) for all new commits.
- Recommended types: `feat`, `fix`, `docs`, `refactor`, `chore`, `test`.
- Write messages in clear, human-readable language.
- Keep the subject understandable enough to be reused directly as changelog text.
